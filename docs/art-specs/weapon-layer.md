# Weapon layer — art spec and engine contract

Decided 2026-07-21 (claude.ai art session). Stage 1 is shipped; stage 2 is
the engine work this doc specifies.

## Why

Dustin's call: the equipped weapon must be visible in the player's hand, and
the swing must read as a real swing (anticipation → arc → contact → follow-
through), not a static rectangle. Weapons are already data (`WeaponData`
.tres) — the art has to follow: one body animation set, any weapon rendered
in hand.

## Stage 1 — SHIPPED (baked pick, new 40x56 sheet)

- `tools/sprites/player_frames.py` is the Delver redesign: 40x56 canvas at
  **1x scale** (PlayerSprite scale is now `Vector2(1, 1)` in player.tscn;
  same on-screen size as the old 20x28 @ 2x).
- The default pickaxe is still drawn INTO the frames by `_pick()`, anchored
  to the glove per pose, so the game looks right with zero engine changes.
- The attack is now 5 poses: wind → overhead → launch → contact → follow.
  `player_sprite.gd` already fits one cycle to the weapon's real
  startup+active+recovery, so nothing there changes.

## Stage 2 — TO BUILD (WeaponSprite node)

The baked pick comes OUT of the body frames (`_POSES[...] show -> False` /
delete the `_pick` call) and a `WeaponSprite` renders the equipped weapon.

### The manifest contract (already being written)

`assets/sprites/player.json` now carries an `anchors` block, one entry per
frame of every animation:

```json
"anchors": {
  "attack": [
    {"hand": [27, 14], "angle": 140, "show": true},
    ...
  ]
}
```

- `hand`: pixel coords of the glove in the 40x56 frame, origin top-left.
  Convert to node space with the Sprite's offset: `pos = hand + (-20, -56)`.
- `angle`: weapon shaft angle in degrees, 0 = forward (+x), 90 = straight
  up. Godot rotation = `deg_to_rad(-angle)` (y is down), flip with facing.
- `show`: false means holster the weapon entirely (roll).

### WeaponSprite (new node under VisualRoot, sibling after Sprite)

- `Sprite2D` with the weapon's texture from `WeaponData` (new export:
  `sprite: Texture2D`, plus `grip: Vector2` — the pixel in the weapon
  texture that sits on the hand anchor).
- Every visual frame (`_process`): read PlayerSprite's current animation +
  frame index, look up the anchor entry, set position/rotation/visible.
  Mirror x when `player.facing < 0` (flip_h + negate offset.x, same as
  PlayerSprite does).
- Draws AFTER the body (higher in tree order) except the `wind` pose where
  the shaft should pass behind the head — if that reads wrong in game, add
  `show_behind: bool` to the anchor entries and toggle z_index. Don't build
  it until it's needed.

### Weapon sprites

New sheet `assets/sprites/weapons.png` + manifest, generated by a
`tools/sprites/weapon_frames.py` (ASCII, same discipline): pickaxe, dagger,
maul, spear. Each ~24x24 with a declared grip pixel. Drawn shaft-up at
angle 90, so rotation math stays trivial. (Authored in the claude.ai art
session; do not hand-draw.)

### Swing feel (engine, not sprites)

- Arc smear: a short-lived `Line2D`/`Polygon2D` trail swept between the
  wind and contact angles around the hand anchor, amber, 2-frame life.
  This replaces baked smear frames — it scales to any weapon length and
  speed for free.
- Contact frame already pairs with hit_spark + hitstop; no change.

### Checks

- Headless check + feel test must pass; visually verify one full swing per
  weapon in the gym.
- The anchors block is additive — PlayerSprite ignores it, so stage 2 can
  land without touching the sheet again.
